# Notion 课题项目保存技能

## 概述

本技能用于将 Claude 对话中的完整输出保存到用户的 Notion 课题项目中。核心原则：

1. **完整保存**：绝不精简、截断、总结任何内容。用户说的"不要精简"是铁律。
2. **智能匹配**：自动探测课题项目中的所有 database，根据内容类型选择最合适的 database。
3. **属性自动填充**：读取目标 database 的属性结构（schema），智能填写标题、标签、日期、状态等属性。
4. **格式保留**：Markdown 格式、代码块、表格、列表等全部保留。

---

## 前置条件

- 用户已连接 Notion MCP 连接器（`mcp.notion.com`）
- 如果未连接，提示用户先连接 Notion

---

## 执行流程

### Step 1: 确认保存内容

确认要保存的内容范围。默认保存 **上一条 Claude 回答的完整内容**。如果用户指定了其他范围（如"保存这整段对话"、"保存最近三轮对话"），按用户指定的范围收集内容。

**关键**：收集内容时，必须保留所有原文，包括：
- 完整的文字段落（不做摘要）
- 代码块（保留语言标记和完整代码）
- 表格（保留完整结构）
- 列表和编号（保留层级）
- 标题和分隔线
- 引用块
- 如果有多轮对话，以 `---` 分隔每轮，并标注 `**User:**` 和 `**Claude:**`

### Step 2: 搜索课题项目

使用 Notion MCP 的 `search` 工具搜索用户的课题项目页面：

```
调用 Notion search 工具，搜索关键词："课题" 或项目名称
```

搜索策略（按优先级）：
1. 如果用户指定了项目名称 → 直接搜索该名称
2. 如果用户说"课题项目" → 搜索 "课题"
3. 如果用户说"研究项目" → 搜索 "研究" 或 "Research"
4. 如果都不明确 → 搜索 "课题" "项目" "Research" "Project" 等关键词
5. 如果搜索结果有多个匹配项 → 列出让用户选择

**搜索无结果时的降级处理**：

如果所有关键词搜索均未找到课题项目页面，执行以下降级流程：

1. **提示用户**：告知未在 Notion 中找到课题项目，并说明可选方案
2. **提供选项**（让用户选择）：
   - **选项 A：搜索 Notion 中已有的其他页面/Database** — 使用 `search` 工具以更宽泛的关键词搜索用户 workspace 中的任意 database 或页面，列出找到的结果让用户选择保存目标
   - **选项 B：直接保存为独立页面** — 使用 `create-pages` 在 workspace 根目录下创建一个独立页面（无 database 属性），直接将内容写入页面
   - **选项 C：放弃保存** — 用户决定暂不保存
3. **根据用户选择执行**：
   - 选 A → 搜索并列出 workspace 中的 database，用户选定后跳转到 Step 4（分析属性结构）
   - 选 B → 跳转到 **"无 Database 的独立页面保存流程"**（见下方）
   - 选 C → 结束，不执行保存

#### 无 Database 的独立页面保存流程

当用户选择保存为独立页面（不关联任何 database）时：

1. 确定页面标题（从内容中提取主题，或询问用户）
2. 使用 `create-pages` 工具创建页面：
   - `parent`：省略（创建为 workspace 级别的私有页面）
   - `properties`：仅包含 `title` 属性
   - `content`：完整内容以 Notion Markdown 格式写入
3. 创建成功后反馈用户，并建议后续可以在 Notion 中手动将页面移到合适的位置

### Step 3: 获取项目结构，发现所有 Database

找到课题项目页面后，使用 `fetch` 工具获取项目页面的内容和子页面结构：

```
调用 Notion fetch 工具，获取项目页面详情
```

从返回结果中识别所有 database（inline database 和 linked database 都要检查）。通常课题项目可能包含的 database 类型：

| 常见 Database 名称 | 典型内容 |
|---|---|
| 文献库 / References / Papers | 论文、参考文献、文献笔记 |
| 写作 / Writing / Drafts | 论文章节、草稿、写作片段 |
| 数据分析 / Data / Analysis | 数据分析记录、统计结果、代码 |
| 进度 / Progress / Tasks | 任务、里程碑、会议记录 |
| 笔记 / Notes / Memos | 通用笔记、思考、灵感 |
| AI对话 / AI Chats / Claude记录 | AI 辅助记录 |

### Step 4: 分析 Database 属性结构

对每个发现的 database，使用 `fetch` 工具获取其属性 schema：

```
调用 Notion fetch 工具，获取每个 database 的结构
```

记录每个 database 的：
- **Database ID**
- **Database 标题**
- **属性列表**（每个属性的名称、类型、选项值）

属性类型映射参考：
- `title` → 页面标题
- `rich_text` → 文本内容、摘要
- `select` / `multi_select` → 分类标签、状态
- `date` → 日期
- `checkbox` → 布尔标记
- `url` → 链接
- `number` → 数值
- `relation` → 关联其他 database
- `status` → 状态（如 Not started / In progress / Done）
- `people` → 负责人
- `created_time` / `last_edited_time` → 时间戳

#### 匹配规则

1. **内容特征 → Database 匹配**  

| 内容特征 | 匹配 Database | 匹配信号 | 属性使用 |
|-----------|---------------|----------|-----------|
| 文献综述、论文笔记 | 领域综述与前沿追踪 | 含 DOI / 作者 / 期刊 / 年份 | 填充组学类型 / 内容类型 / 进度状态 / 其他标签 |
| 数据库资源整理、模型数据 | 现有模型和数据 | 含数据库、文件链接、资源 | 填充组学类型 / 内容类型 / 进度状态 / 其他标签 |
| 代码、算法实现、工具开发 | 模型构建算法 / 编程与工具 | 含 algorithm / code / 工具教程 / GEM / FBA | 填充组学类型 / 内容类型 / 进度状态 / 其他标签 / 疾病类型 |
| AI 对话 / 课题想法 | 课题思路 | 含 idea / 策略 / 研究方案 | 填充内容类型 / 进度状态 |
| 无明确分类 | 编程与工具 或 笔记库 | 通用文本 / 技术笔记 | 填充组学类型 / 内容类型 / 进度状态 |

2. **匹配流程**：
   - 扫描对话或文档关键词：
     - “数据库”“模型”“资源” → 现有模型和数据
     - “算法”“GEM”“通量分析” → 模型构建算法
     - “综述”“前沿”“review” → 领域综述与前沿追踪
     - “代码”“脚本”“工具” → 编程与工具
     - “idea”“思路”“计划”“AI” → 课题思路
   - 对每个 database 的属性 schema 打分（关键词匹配度）
   - 匹配度最高 → 自动选择
   - 若多个 database 匹配度接近 → 提示用户选择
   - 如果只有一个 database → 直接使用

### Step 6: 准备属性值

根据目标 database 的属性 schema，自动准备属性值：

**标题属性（title）**：
- 如果内容有明确主题 → 提取主题作为标题
- 格式建议：`Claude: 内容主题`
- 示例：`Claude: 微塑料文献综述框架`

**日期属性（date）**：
- 填入当天日期

**标签/分类属性（select / multi_select）**：
- 先检查 database 已有的选项值
- 从已有选项中匹配最合适的
- 如果没有完全匹配的，选择最接近的
- 如果有"AI生成"、"Claude"等选项，勾选上
- 多选标签：根据内容涉及的多个主题选择

**状态属性（status）**：
- 默认选择 "未精读"
- 如果没有完成类选项，选择最合适的中间态

**来源/类型属性**：
- 如果有"来源"属性 → 填入 "Claude AI" 或 "AI 对话"
- 如果有"类型"属性 → 根据内容填写（如"文献综述"、"数据分析"、"写作草稿"）

**其他属性**：
- URL → 如果内容中有相关链接，填入
- 关联（relation）→ 暂时跳过，除非用户指定
- 人员（people）→ 跳过
- 未知用途的属性 → 跳过（宁缺毋滥）

### Step 7: 创建 Notion 页面

使用 Notion MCP 的 `create-pages` 工具创建新页面：

```
调用 Notion create-pages 工具：
- parent: 目标 database ID
- properties: Step 6 中准备的属性值
- children: 完整内容转换为 Notion block 格式
```

**内容格式转换规则**：

将 Markdown 内容转换为 Notion blocks 时：
- `# 标题` → heading_1 block
- `## 二级标题` → heading_2 block
- `### 三级标题` → heading_3 block
- 普通段落 → paragraph block
- `` `代码块` `` → code block（保留语言标记）
- `- 列表` → bulleted_list_item block
- `1. 编号` → numbered_list_item block
- `> 引用` → quote block
- `---` → divider block
- 表格 → table block
- 图片链接 → image block

**⚠️ 重要：Notion API 单次创建的 children blocks 有数量限制**。如果内容很长：
1. 先用 `create-pages` 创建页面（含属性和前面部分内容）
2. 如果有剩余内容，提醒用户内容可能需要分次添加
3. 考虑将超长内容拆分为多个段落 block

### Step 8: 确认并反馈

创建成功后，向用户反馈：
- ✅ 已保存到：`[Database 名称]`
- 📄 页面标题：`[标题]`
- 🏷️ 已填属性：列出已填写的属性及值
- 🔗 提供 Notion 页面链接（如果 API 返回了链接）

如果创建失败：
- 显示错误信息
- 建议可能的解决方案（如权限问题、属性格式错误等）
- 提供手动保存的替代方案

## 特殊场景处理

### 场景 A：课题项目中没有合适的 Database
- 询问用户是否要在项目中新建一个 database
- 如果是，建议一个合适的 database 结构（名称 + 基础属性）
- 使用 `create-database` 工具创建
- 然后继续正常保存流程

### 场景 B：用户要保存多轮对话
- 以分隔线区分每轮对话
- 每轮标注角色（User / Claude）
- 保留所有轮次的完整内容
- 标题反映对话的整体主题

### 场景 C：内容包含代码
- 代码必须完整保留在 code block 中
- 标注编程语言
- 如果代码很长，不要截断

### 场景 D：内容包含表格或数据
- 尽量用 Notion table block 保留表格结构
- 如果表格格式无法完美转换，使用代码块作为备选
- 数值和统计结果必须精确保留

### 场景 E：用户要求保存到特定 Database
- 用户可能说"存到文献库"、"存到笔记里"
- 直接匹配用户指定的 database，跳过智能匹配
- 仍然执行属性自动填充

### 场景 F：Notion 中没有课题项目
- 按照步骤 2 的"搜索无结果时的降级处理"流程执行
- 提供三个选项：搜索已有 database、保存为独立页面、放弃保存
- 如果用户选择搜索已有 database，使用宽泛搜索找到 workspace 中的可用目标
- 如果用户选择独立页面，跳过属性填充，直接创建 workspace 级别的页面
- 在反馈中提醒用户：后续可以在 Notion 中将页面移入合适的课题项目或 database

---

## 注意事项

1. **绝不精简内容**：这是本技能的核心原则。无论内容多长，都完整保存。
2. **属性宁缺毋滥**：不确定的属性值宁可不填，也不要填错。
3. **尊重已有结构**：不要修改用户 database 的现有属性，只使用已有的属性。
4. **保护隐私**：如果内容包含敏感信息，提醒用户确认是否要保存。
5. **错误处理要友好**：Notion API 可能有各种限制，遇到问题要给出明确的解释和替代方案。
6. **日期时区**：使用用户当地时区的日期。
